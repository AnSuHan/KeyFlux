# 키 입력 처리와 블로킹 정책

KeyFlux 가 전역 키보드 후크로 키를 어떻게 다루는지, 그리고 **어떤 경우에 키를
막고(block) 어떤 경우에 그대로 흘려보내는지**를 정리한 문서다. 관련 코드는 모두
`main.py` 의 `KeyboardListener` 클래스에 있다.

---

## 1. 기본 원칙 — 평상시엔 아무 키도 막지 않는다(관찰 전용)

전역 후크는 `pynput` 의 `keyboard.Listener` 로 설치하며, **`suppress=False`** 로
띄운다(`main.py:1322`). 즉 리스너는 키 이벤트를 **관찰(observe)만** 하고, 어떤
키도 가로채거나 삼키지 않는다. 등록된 트리거가 완성됐을 때만 백스페이스 +
재입력으로 치환할 뿐이다.

근거(“평상시엔 Enter·Insert 등 어떤 명령키도 막지 않는다”):

- **후크가 non-suppress** (`main.py:1322`, `suppress=False`).
  `pynput` 에서 전역 억제가 켜지려면 `suppress=True` 여야 한다.
- **`_on_press` 가 절대 `return False` 하지 않는다** (`main.py:956`).
  `pynput` 콜백은 `False` 를 돌려줘야 그 키가 막히는데, `_on_press` 는 모든
  분기에서 `None`(암묵)만 반환한다 → 개별 키 단위 차단도 없음.
  - `Key.enter` / `Key.space` → 매칭 검사만 하고 통과
  - `Key.backspace` → 내부 버퍼만 한 칸 줄이고 통과
  - 그 외 특수키(**Insert**, 방향키, Delete, Ctrl 등) → 트리거 조합 버퍼만
    초기화하고 통과. 즉 Insert 등은 “트리거 조합을 끊는” 신호로만 쓰이고,
    **명령 자체는 대상 앱에서 정상 동작**한다.
- **`_suppress_until` / `_replacing` 은 OS 차단이 아니다.**
  이 값들은 치환 주입 중 “관찰자”가 자기 합성 입력을 트리거 버퍼에 다시
  기록하지 않게 하는 내부 부기(bookkeeping)일 뿐, OS 로의 전파를 막지 않는다.
- **Enter 로 완성되는 word/regex 트리거도 Enter 를 삼키지 않는다.**
  사용자가 친 스페이스/엔터는 이미 화면에 찍혀 있고(`extra`), 치환 시
  백스페이스로 지운 뒤 `출력 + extra` 로 되찍어 복원한다(막는 게 아니라 재현).

## 2. 예외 — “변환(치환) 진행 중”에는 사용자 실입력을 전역 차단한다

키워드가 완성되어 **‘키워드 → 출력’ 치환이 주입되는 짧은 동안**에는, 사용자가
그 사이에 누른 **한/영·Enter·Backspace 등 실제 키 입력을 시스템 전역으로
차단**한다. 이렇게 하지 않으면 주입 중인 백스페이스·텍스트와 사용자 입력이
뒤섞여 결과가 깨질 수 있다. **변환이 끝나면 차단은 즉시 풀린다.**

### 2.1 어떻게 차단하는가 (Windows 저수준 후크 필터)

`pynput` 의 `win32_event_filter(msg, data)` 훅 필터를 등록한다
(`main.py:1298`, `start()` 에서 win32 에서만 연결 — `main.py:1326`).

- 필터 안에서 **`self._listener.suppress_event()`** 를 호출하면
  `SuppressException` 이 발생한다.
- 이 예외는 `pynput` 의 저수준 후크(`SystemHook`)가 잡아 **`CallNextHookEx` 를
  호출하지 않고 `return 1`** 하게 만든다 → 그 키 이벤트가 대상 앱/시스템으로
  **전파되지 못하고 전역 차단**된다.
- 그래서 필터 코드에서 이 예외를 절대 잡으면 안 된다(잡으면 차단이 안 됨).

> 참고: 필터가 단순히 `False` 를 반환하면 “리스너 콜백(on_press)에만 전달 안 함”
> 이고 **OS 차단은 아니다**. 실제 차단은 위의 `suppress_event()` 경로로만 된다.

### 2.2 “우리가 주입한 이벤트”는 차단에서 제외한다

치환은 우리가 직접 백스페이스·텍스트를 **주입(SendInput)** 해서 이뤄진다. 이
주입 이벤트까지 막아버리면 치환 자체가 스스로 막힌다. 그래서 후크 구조체의
**주입 플래그 `LLKHF_INJECTED`(0x10)** 로 구분한다(`main.py:1277`).

- `data.flags & 0x10` 이 서 있으면 → 우리(또는 시스템) 주입 이벤트 → **통과**.
- 플래그가 없으면(=사용자가 물리 키보드로 친 실제 입력) → 변환 중이면 **차단**.

Windows 는 `SendInput` 으로 넣은 이벤트에 자동으로 `LLKHF_INJECTED` 를 세우므로,
우리 백스페이스(`self.controller` 경유)와 유니코드/스캔코드 텍스트 주입은 모두
이 플래그가 서 있어 차단에서 자연히 제외된다.

### 2.3 “변환 진행 중” 창의 정의

`_blocking_now()`(`main.py:1279`)가 아래 **세 조건을 모두** 만족할 때만 차단한다.

1. 리스너가 활성 상태(`self.active`).
2. 치환 주입이 진행 중 —
   주입 스레드 실행 중(`self._replacing`, `main.py:1141`에서 set) **또는**
   억제 시간 이내(`time.monotonic() < self._suppress_until`, `main.py:1182`).
3. 이벤트가 **주입 플래그가 없는** 사용자 실입력.

세 조건 중 하나라도 아니면 차단하지 않는다. `_replacing` 은 주입이 실제 끝나는
`_inject_replacement`(`main.py:1209`)의 `finally` 에서 해제되고, `_suppress_until`
은 예상 소요 시간 뒤 자동 만료된다 → **추정이 틀려도 변환이 끝나면 반드시
차단이 풀린다.**

## 3. 한눈에 보기

| 상황 | 처리 |
|---|---|
| 평상시(치환 안 함) 모든 키 | **통과** (관찰만, `suppress=False`) |
| 치환 진행 중 · 사용자 실입력(한/영·Enter·Backspace 등) | **전역 차단** |
| 치환 진행 중 · 우리 주입 이벤트(백스페이스·텍스트) | **통과** (LLKHF_INJECTED 제외) |
| 리스너 비활성(일시정지) | **통과** |

## 4. 코드 참조 (main.py)

| 위치 | 역할 |
|---|---|
| `start()` (≈1313) | 리스너 생성, win32 에서 `win32_event_filter` 연결 |
| `_win32_event_filter()` (≈1298) | 후크 필터. 차단 대상이면 `suppress_event()` 호출 |
| `_blocking_now()` (≈1279) | 차단 여부 판단(활성·변환중·비주입) |
| `_LLKHF_INJECTED = 0x10` (≈1277) | 주입 이벤트 식별 플래그 |
| `_do_replace()` (≈1135) | `_replacing`/`_suppress_until` set 후 주입 스레드 시작 |
| `_inject_replacement()` (≈1209) | 실제 백스페이스+텍스트 주입, 종료 시 `_replacing` 해제 |
| `_on_press()` (≈956) | 관찰 콜백(절대 `return False` 안 함) |

> 라인 번호는 참고용이며, 코드 수정에 따라 다소 달라질 수 있다.
